Learning Outcomes

  • Model systems with M/M/1, M/D/1, M/M/1/K, and M/M/K (Erlang-C).
  • Translate arrival and service rates into latency predictions and back.
  • Design experiments: choose representative workloads and avoid benchmarking traps.
  • Set and validate SLOs (e.g., “p95 < 300 ms”) with proper SLIs and alerting.
  • Use results to make capacity and cost trade-offs.

Quick Reference

Little’s Law

L = λ · W
Lq = λ · Wq

Holds for stable systems; sanity-check your measurements with it.

M/M/1 formulas

ρ  = λ / μ          (ρ < 1)
W  = 1 / (μ − λ)
Wq = ρ / (μ − λ)

As ρ → 1, queueing explodes; keep planned peaks ≤ 0.7–0.8.

Erlang-C (M/M/K)

a = λ/μ,  ρ = a/K
Pwait = [ (a^K / K!) · (K / (K − a)) ]
        / [ Σ_{i=0}^{K−1} (a^i / i!) + (a^K / K!) · (K / (K − a)) ]
Wq    = Pwait / (K·μ − λ)
W     = Wq + 1/μ

Use to size replicas K for a target W (e.g., p50 budget) with headroom.

Rules of thumb

  • Parallelism beats buffering for latency SLOs.
  • Stabilise service time: variance drives queueing.
  • Alert near the “knee”: W ≈ service time.

Lab Snippet — Simple Load Test

# Using 'hey' (Go HTTP load generator)
hey -z 60s -q 0 -c 64 -m GET https://localhost:8080/api

# Using wrk
wrk -t8 -c128 -d60s --latency https://localhost:8080/api

Warm up the system first; capture percentiles and CPU/memory/I/O metrics alongside RPS.

Key Takeaways

  • Use models to predict and experiments to verify under real workloads.
  • Design to percentiles, not averages; tails drive user experience.
  • Keep utilisation modest at peak; small headroom prevents big tail spikes.
