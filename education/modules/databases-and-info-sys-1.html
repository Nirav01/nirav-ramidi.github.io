Learning Outcomes

  • Model a problem domain with ER diagrams and transform to a relational schema.
  • Write correct SQL queries (selection, joins, grouping/aggregation, subqueries).
  • Apply normal forms (1NF–3NF/BCNF) to remove redundancy and anomalies.
  • Explain and use transactions (ACID) and understand isolation phenomena.
  • Choose indexes to improve performance; reason about query plans.

Quick Reference

SQL pattern — joins & grouping

SELECT a.name, COUNT(o.id) AS orders
FROM accounts a
LEFT JOIN orders o ON o.account_id = a.id
WHERE a.active = TRUE
GROUP BY a.name
HAVING COUNT(o.id) >= 1
ORDER BY orders DESC
LIMIT 10;

Prefer explicit JOIN syntax; group by selected non-aggregates.

Normal forms (at a glance)

1NF: atomic columns, no repeating groups
2NF: 1NF + no partial deps on a composite key
3NF: 2NF + no transitive deps on key
BCNF: every determinant is a candidate key

Decompose when FDs violate 3NF/BCNF to avoid update anomalies.

Transactions & isolation

BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id=1;
UPDATE accounts SET balance = balance + 50 WHERE id=2;
COMMIT;  -- or ROLLBACK;

Use the weakest isolation that preserves correctness; beware lost updates & phantom reads.

Indexes (Postgres)

-- Single & composite indexes
CREATE INDEX idx_orders_account ON orders(account_id);
CREATE INDEX idx_orders_status_created ON orders(status, created_at);

-- Inspect plan
EXPLAIN ANALYZE
SELECT * FROM orders WHERE status='shipped' AND created_at>now()-interval '30 days';

Put most selective columns first in composite indexes; avoid redundant overlaps.

Key Takeaways

  • Design schemas from real-world entities and constraints before writing queries.
  • Normalise to remove anomalies; denormalise deliberately with a performance goal.
  • Read query plans; indexes are tools, not magic—measure before/after.
