Java Marketplace

A full-stack Spring Boot marketplace with catalogue, cart/checkout, authentication & roles, order history, and an admin back-office. Built with clean MVC, form validation, and integration tests.

  • Spring Boot · Thymeleaf
  • REST + MVC
  • PostgreSQL (H2 dev)
  • Spring Security (BCrypt)

Problem & Goals

Build a small but realistic online store with user accounts, secure checkout, and basic admin operations. The emphasis is on clean domain modelling, secure authentication, and a testable MVC architecture.

  • Catalogue with categories, search, and product detail pages.
  • Cart → checkout flow with address & payment stub.
  • Authentication/authorisation: USER vs ADMIN.
  • Admin back-office for CRUD on products and price history.

Architecture

Domain Model

  • User (id, email, passwordHash, roles)
  • Product (id, sku, name, description, category, PriceHistory)
  • Cart / CartItem (session or user-bound)
  • Order / OrderItem (immutable totals, snapshot of prices)
  • PriceHistory (product, price, effectiveFrom, effectiveTo)

Persistence

  • Spring Data JPA + PostgreSQL (H2 for local/dev).
  • Flyway migrations to version schema.
  • Eager vs LAZY tuned per aggregate (e.g., order → items).

Views & API

  • Server-rendered views via Thymeleaf for store & admin.
  • Small JSON endpoints for cart actions (AJAX).

Security

  • Spring Security with form login and role-based access control.
  • BCrypt password hashing; users seeded via migration with hashed secrets.
  • CSRF tokens on state-changing requests; input validation & bean validation.
  • Restricted admin routes under /admin/**; method-level guards for mutations.

Key Features

Catalogue

Browse by category, keyword search, and product detail with related items.

Cart & Checkout

Add/remove items, address capture, payment stub, order confirmation email stub.

Order History

Users can review past orders; totals are immutable from price snapshots.

Admin Back-Office

CRUD products, manage categories, and maintain PriceHistory with effective dates.

Testing

  • Unit tests for services and domain rules (JUnit 5 + Mockito).
  • Web layer tests with MockMvc (auth, CSRF, form validation).
  • Repository tests against H2 with Flyway to validate mappings and queries.

What I learned

  • Enforcing immutable order totals requires price snapshotting and careful mapping.
  • Clean separation of concerns simplifies testing and refactoring.
  • Security defaults (CSRF, password hashing) are easy to miss unless tested explicitly.
