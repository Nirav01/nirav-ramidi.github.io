Huffman Compressor

A from-scratch Java implementation of Huffman coding with optimal prefix codes, bit-level I/O, and canonical code serialisation. Includes a CLI for compress/decompress and a benchmark harness over mixed corpora.

  • Greedy + Priority Queue
  • Canonical codes
  • BitOutputStream
  • Unit & property tests

Problem & Goals

Implement a lossless compressor using Huffman coding, ensuring portability and correctness: deterministic codebooks, robust bit-level read/write, and linear-time encode/decode after codebook build.

  • File format with self-contained header (code lengths) for decoder reconstruction.
  • Support full 8-bit byte alphabet (σ ≤ 256) and arbitrary binary input.
  • Provide a CLI with helpful errors and progress stats.

Algorithm

Build (one pass over frequencies)

  1. Count symbol frequencies → array[0..255].
  2. Push non-zero symbols into a min-heap by frequency.
  3. Pop two, merge to parent with sum freq, push back — repeat.
  4. Derive code lengths from root-to-leaf depths.

Time: O(σ log σ + n) with σ≤256 ⇒ effectively O(n). Space: O(σ).

Canonicalisation

  • Sort (length, symbol) to produce canonical codes — compact header, deterministic.
  • Decoder reconstructs codes from lengths only (no tree serialisation needed).

I/O

  • BitOutputStream / BitInputStream buffer bits to/from bytes.
  • Pad final byte; record payload bit length in header.

Implementation Highlights

Header format

Magic bytes, version, 256 code lengths (0..maxLen), and payloadBits. Decoder rebuilds canonical codes from lengths; zero length ⇒ symbol absent.

Streaming safety

Chunked frequency count to cap memory; encoder writes as it reads, no full file buffering.

Edge cases

Empty input; single-symbol files (emit 1-bit code); binary data with all 256 symbols; corrupted header checks.

Testing

Unit tests for heap/tree, canonical codes, and round-trip decode(encode(x)) = x. Property tests across random byte arrays; golden files for CLI.

Benchmarks (indicative)

Ran on text (UTF-8), logs, and small binaries. Huffman excels on skewed text; header overhead impacts tiny files. General-purpose codecs (e.g., DEFLATE) may beat Huffman on mixed/binary due to LZ stages, but Huffman remains a great teaching-quality baseline with predictable behaviour.

  • Compression time roughly linear in input size after codebook build.
  • Compression ratio improves with symbol skew and larger files.
  • Canonical header reduced file-format size vs raw tree serialisation.

What I learned

  • How greedy tree construction yields an optimal prefix code.
  • Why canonical codes simplify serialisation and speed decoding.
  • The importance of deterministic formats and bit-precise I/O.
